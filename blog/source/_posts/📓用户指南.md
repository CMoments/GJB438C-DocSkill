# 用户使用指南

> word-fillter-438C 详细操作指南

## 1. 概述

### 1.1 关于本指南

本指南面向使用 word-fillter-438C 生成软件需求规格说明书的用户，提供详细的操作步骤和最佳实践。

### 1.2 目标读者

- 软件工程师
- 系统分析师
- 项目经理
- 质量保证人员

### 1.3 前置条件

| 条件 | 说明 |
|------|------|
| Claude Code | 已安装 Claude Code CLI |
| Python 3.8+ | 已配置 Python 运行环境 |
| 必要依赖 | python-docx |

## 2. 快速开始

### 2.1 启动 Skill

在 Claude Code 中，通过以下方式调用此技能：

```
按照438C模板填写word文档
```

或者简单地描述你的需求：

```
帮我生成一份软件需求规格说明书
```

### 2.2 工作流程概览

```
┌──────────────────────────────────────────────────────────┐
│  Step 1: 填写项目信息 (config.json - project 部分)        │
├──────────────────────────────────────────────────────────┤
│  Step 2: 填写文档信息 (config.json - document 部分)       │
├──────────────────────────────────────────────────────────┤
│  Step 3: 填写详细需求信息 (project.json)                  │

```

## 3. 详细操作步骤

### 3.1 Step 1: 填写项目信息

系统会通过交互式问答收集项目信息，这些信息将用于生成 `config.json` 的 `project` 部分。

#### 3.1.1 项目信息字段

| 字段 | 说明 | 示例 |
|------|------|------|
| overview | 项目概述 | 本项目用于开发一套智能仓储管理系统... |
| scope | 项目范围 | 包含入库管理、出库管理、库存盘点模块 |
| background | 项目背景 | 由于业务发展需要，原有系统无法满足... |
| objectives | 项目目标 | 提升仓储效率30%、降低人工成本 |
| features | 功能特性 | 实时库存查询、自动预警、数据分析 |
| target_users | 目标用户群体 | 仓库管理员、采购人员、管理层 |
| constraints | 项目约束 | 需在Q2上线、预算上限50万 |
| assumptions | 假设与依赖 | 假设服务器资源充足、依赖第三方API |
| references | 参考资料 | 《仓储管理规范》、GB/T 8567-2006 |
| terminology | 术语定义 | SKU-库存单位、WMS-仓库管理系统 |

#### 3.1.2 填写技巧

- **简洁明了**：使用简短但完整的描述
- **保持一致**：项目名称、术语在后续步骤中保持一致
- **遵循规范**：参考资料应注明来源
- **准确定义**：术语定义应清晰无歧义

### 3.2 Step 2: 填写文档信息

继续通过问答收集文档封面信息，生成 `config.json` 的 `document` 部分。

#### 3.2.1 文档信息字段

| 字段 | 说明 | 示例 |
|------|------|------|
| 密级 | 文档密级 | 内部（可选：内部/公开/机密） |
| 阶段 | 开发阶段 | 设计阶段 |
| 版次 | 文档版次 | A 版 |
| 外部型号+产品名称 | 产品完整名称 | SAMPLE-智能终端 |
| 产品型号 | 产品型号标识 | SAMPLE-V1.0 |
| 公司名称 | 开发单位名称 | XXXXXXXX公司 |
| 日期 | 文档日期 | 2026年03月28日 |

#### 3.2.2 密级说明

| 密级 | 适用场景 |
|------|----------|
| 公开 | 可对外发布的文档 |
| 内部 | 仅限公司内部使用 |
| 机密 | 涉及核心技术或商业秘密 |

### 3.3 Step 3: 填写详细需求信息

系统会继续通过交互式问答收集详细需求信息，生成 `project.json`。

#### 3.3.1 章节结构

`project.json` 遵循 GJB 438C 标准的章节结构：

| 章节 | 标题 | 内容类型 |
|------|------|----------|
| 1 | 范围 | 标识、系统概述、文档概述 |
| 2 | 引用文档 | 引用的标准和规范（表格） |
| 3 | 需求 | 状态方式、能力需求、接口需求等 |
| 4 | 合格性规定 | 需求验证方法（表格） |
| 5 | 需求可追踪性 | 需求追踪矩阵（表格） |
| 6 | 注释 | 术语和缩略语 |

#### 3.3.2 字段类型说明

| 类型 | 说明 | 填写方式 |
|------|------|----------|
| original | 直接填入用户输入的内容 | 问答获取 |
| text_generation | 根据描述生成文本 | 结合问答和描述生成 |
| content_generation | 根据提示生成结构化内容 | 结合问答和提示生成 |

#### 3.3.3 动态章节处理

对于 3.2.X（软件能力需求）和 3.3.X（软件外部接口需求）等动态章节：

1. **系统生成预设子项**：根据 `generation_prompt` 生成初始子项列表
2. **用户确认调整**：可以增加、删除或修改子项
3. **逐项填写内容**：为每个子项填写详细需求

#### 3.3.4 填写示例

**3.2.X 软件能力需求** 的填写：

```
系统提示：请为软件设计3-8个核心功能模块，每个模块包括：功能名称、功能描述、输入、输出、性能要求、约束条件

用户回答：
1. 用户管理模块
   - 功能描述：管理系统用户账号和权限
   - 输入：用户信息、权限配置
   - 输出：操作结果、用户列表
   - 性能要求：响应时间<1秒
   - 约束条件：密码必须加密存储

2. 数据采集模块
   - 功能描述：从传感器采集实时数据
   - 输入：传感器信号
   - 输出：结构化数据
   - 性能要求：采集频率100Hz
   - 约束条件：支持断点续传
```

#### 3.3.5 嵌套子章节

某些章节包含固定的嵌套子章节，如 3.10（计算机资源需求）：

```
3.10 计算机资源需求
├── 3.10.1 计算机硬件需求
├── 3.10.2 计算机硬件资源使用需求
├── 3.10.3 计算机软件需求
└── 3.10.4 计算机通信需求
```

每个子章节需要单独填写内容。

### 3.4 Step 4: 生成 SRS 文档

填写完成后，系统会自动执行文档生成。

#### 3.4.1 执行命令

```bash
python main.py \
  --template "[08]软件需求规格说明-438C.docx" \
  --config "config.json" \
  --project "project.json" \
  --output "output/[08]软件需求规格说明-438C-filled.docx"
```

#### 3.4.2 生成过程

```
┌─────────────────────────────────────────┐
│  1. 加载 config.json 和 project.json    │
├─────────────────────────────────────────┤
│  2. 打开 Word 模板                      │
├─────────────────────────────────────────┤
│  3. 替换封面信息                         │
├─────────────────────────────────────────┤
│  4. 替换正文内容                         │
├─────────────────────────────────────────┤
│  5. 处理动态章节                         │
├─────────────────────────────────────────┤
│  6. 保存输出文档                         │
└─────────────────────────────────────────┘
```

#### 3.4.3 输出文件

- 输入模板：`[08]软件需求规格说明-438C.docx`
- 输出文件：`output/[08]软件需求规格说明-438C-filled.docx`

### 3.5 Step 5: 验证输出文档

最后一步是验证生成的 Word 文档。

#### 3.5.1 验证内容

| 检查项 | 说明 |
|--------|------|
| ZIP结构 | 验证 docx 作为 ZIP 的完整性 |
| XML格式 | 检查 document.xml 语法正确性 |
| 占位符残留 | 检测未替换的占位符 |
| 文件大小 | 警告异常大小的文件 |

#### 3.5.2 验证结果

**成功**：
```
[OK] 文档验证通过！
```

**警告**：
```
警告:
  ⚠ 发现可能的未替换占位符: 3 个
  ⚠ 文件大小较小，可能内容不完整
```

#### 3.5.3 后续操作

首次在 Word 中打开生成的文档时：
1. 按提示刷新目录（右键目录 → 更新域 → 更新整个目录）
2. 检查页码是否正确
3. 检查表格格式是否完整

## 5. 最佳实践

### 5.1 需求编写建议

| 建议 | 说明 |
|------|------|
| 使用明确语言 | 避免模糊表述，如"可能"、"大概" |
| 量化指标 | 尽量使用可度量的指标 |
| 完整性 | 确保覆盖所有必要需求 |
| 一致性 | 需求之间不应矛盾 |
| 可验证性 | 每个需求都应可测试验证 |

### 5.2 文档命名规范

```
[项目简称]-SRS-[版本号]-[日期].docx
```

示例：`SAMPLE-SRS-V1.0-20260328.docx`

### 5.3 版本更新流程

1. 复制上一版本的配置文件
2. 更新版本号和日期
3. 修改变更内容
4. 重新生成文档
5. 验证输出

## 6. 故障排除

### 6.1 常见问题

#### Q1: 配置文件验证失败

**症状**：验证脚本报错

**解决方案**：
1. 检查 JSON 语法是否正确
2. 确保所有必填字段已填写
3. 确认字段值不是 `"string"` 占位符

#### Q2: 文档生成失败

**症状**：`main.py` 执行报错

**解决方案**：
1. 确认模板文件存在
2. 检查 python-docx 是否正确安装
3. 验证配置文件已通过验证

#### Q3: 输出文档格式异常

**症状**：生成的文档格式不正确

**解决方案**：
1. 检查原始模板是否完整
2. 确认替换内容不包含特殊字符
3. 尝试重新生成

#### Q4: 目录未更新

**症状**：生成的文档目录显示错误

**解决方案**：
1. 在 Word 中右键目录
2. 选择"更新域"
3. 选择"更新整个目录"

### 6.2 获取帮助

如遇到无法解决的问题，请：
1. 保存完整的错误日志
2. 记录复现步骤
3. 联系技术支持

## 7. 附录

### 7.1 GJB 438C 标准参考

GJB 438C 是我国军用软件文档编制标准，规定了软件需求规格说明书的结构和内容要求。

### 7.2 术语表

| 术语 | 说明 |
|------|------|
| SRS | Software Requirements Specification，软件需求规格说明书 |
| GJB | 国家军用标准 |
| JSON | JavaScript Object Notation，一种数据交换格式 |
| placeholder | 占位符，模板中待替换的位置 |
| subsection | 子章节，章节的下一级内容 |
